# Strategic Plan

## 依据

本战略以根目录 `AGENTS.md` 和 `game_design_bible.md` 为最高优先级文档：

- `AGENTS.md` 定义工程、平台、Agent 协作、目录和硬性约束。
- `game_design_bible.md` 定义产品定位、玩法边界、美术方向、经济和运营原则。

当两者存在细节差异时，执行更严格或更利于 MVP 落地的一侧：

- 创作者 `meta.json` 的多语言名称按 `AGENTS.md` 执行，必须包含 `en`、`ja`、`zh`。
- 美术方向继续沿用当前森林/RO-like 基底，避免推翻已有资产和主城实现；港口城可作为后续钓鱼或海边区域扩展。
- UGC 小游戏平台是长期护城河，但加速 MVP 窗口只交付“官方示例 + 接口骨架 + 审核闭环”，不追求完整公开创作者生态。

## 战略定位

项目不是传统副本 MMO，也不是单机小游戏合集，而是：

> 一个轻量 2D 像素在线社交世界，用主城、房屋、聊天、表情和小游戏，把玩家留在一个可持续扩展的社区里。

北极星体验：

- 玩家每天 20-60 分钟内能完成聊天、钓鱼、装饰、小游戏和日常奖励。
- 社交优先于战斗，归属感优先于数值压迫。
- 房屋和外观是长期金币消耗核心。
- 官方小游戏先证明接口，创作者小游戏后续扩大内容供给。
- 付费只服务外观、便利和活动，不卖战斗公平性。

## 当前判断

已有基础已经越过“原型”阶段，正在进入“可验证产品骨架”：

- 客户端已有 Godot 主城、HUD、聊天、在线房间、小游戏入口、表情、房屋和钓鱼示例基础。
- 后端已有 Go 网关、Guest auth、WebSocket、presence、chat、room、economy、housing、minigame session 和 creator review 骨架。
- Image 2 资产路线已确立，旧 SVG 正在被 PNG/WebP 替换。
- 文档体系已覆盖接口、部署、主城结构、多人同步、创作者规范和 Studio 进度。

最大缺口不再是“有没有功能”，而是三件事：

1. 主城、聊天、房屋、小游戏之间的玩家闭环还需要产品化。
2. 经济系统需要从接口走向每日产出/消耗可观测。
3. 创作者平台需要保持架构正确，但不能抢走 MVP 的精力。

当前执行节奏采用加速路线，详见 `docs/AcceleratedContentRoute.md`。原 6 个月路线作为安全基准保留，实际推进按 4-6 周形成可测核心、8-12 周进入小范围真实玩家测试来压缩。

## 六大支柱

### 1. 主城是第一产品

主城必须成为所有系统的入口，而不是一张过场地图。

优先做：

- 可走、可停留、可看到别人。
- 可聊天、可发头顶表情、可切频道。
- 可从热点进入钓鱼、房屋、小游戏大厅、商店。
- 用 Image 2 正式像素建筑和道具替换 blockout 热点。

暂缓做：

- 大地图传送链。
- 多城市切换。
- 复杂 NPC 剧情。

### 2. 聊天和表情先做深

轻社交项目的第一留存来自“我看见别人，也能回应别人”。

优先做：

- 公屏、附近、房屋、队伍频道的清晰路由。
- DM/私聊协议骨架。
- 头顶表情统一走 `EmoteCatalog`、`OverheadEmoteBubble`、`IMinigame.request_emote`。
- 聊天节流、敏感词和举报预留。

暂缓做：

- 复杂语音。
- 大型公会频道。
- 跨服频道。

### 3. 房屋是归属感和金币消耗核心

房屋不是装饰附属功能，而是长期留存容器。

优先做：

- 玩家花金币购买/摆放家具。
- 房间布局和样式由玩家自行建造。
- 好友访问和房屋聊天预留。
- 家具作为金币主要 sink。

暂缓做：

- 高复杂度建造编辑器。
- 家具交易市场。
- 多楼层豪宅。

### 4. 钓鱼是官方标准小游戏

钓鱼不仅是玩法，也是 `IMinigame`、经济、审核、结算和 UI 规范的第一示范。

优先做：

- 客户端钓鱼体验闭环。
- 服务端可信奖励、幂等结算、每日上限。
- 三语 UI key。
- 作为创作者文档中的官方 reference package。

暂缓做：

- 过多鱼种。
- 复杂装备词条。
- PvP 或竞赛化钓鱼。

### 5. 创作者平台先立接口，不急公开

长期要做 UGC，但短期只做可控 alpha。

优先做：

- `IMinigame` 接口稳定。
- `meta.json` 校验稳定。
- 资源体积、API 禁用、节点访问、网络权限扫描。
- 后端异步审核状态机。
- 管理员白名单发布。

暂缓做：

- 玩家自助公开商店。
- 创作者收益提现。
- 复杂版本市场。

### 6. Image 2 资产进入生产管线

Image 2 不是参考图来源，而是正式 UI/美术资产来源。

优先做：

- 生成 sheet 后切图/atlas。
- 正式 PNG/WebP 进入 `assets/`。
- 资源登记到 `configs/ui_assets.json` 或 `configs/art_assets.json`。
- 删除运行时旧 SVG 引用。

暂缓做：

- 大规模角色套装。
- 多城市全量 tileset。
- 全职业战斗表现。

## 加速路线

### Phase 0：主城核心锁定（Week 0-1）

目标：让主城成为所有核心系统的稳定入口。

- 主城热点从 blockout 换为 Image 2 正式建筑/道具。
- 聊天频道、在线房间、小游戏入口、presence 可视化统一到 HUD。
- `IMinigame`、creator manifest、后端 API 文档按新版 `AGENTS.md` 对齐。
- 保持 Godot 单文件小于 300 行，继续拆分 UI/网络脚本。
- 跑通 Godot smoke、Go unit/e2e、Web/H5 viewport smoke。

### Phase 1：热闹小镇内容包（Week 2）

目标：让玩家在主城里“看见、说话、回应、停留”，并且一眼看懂社交、钓鱼、房屋、小游戏入口。

- NPC 第一批：渔夫、商人、邮箱员、小游戏主持、房屋管理员、活动公告员。
- 主城氛围：灯、花、路牌、摊位、邮箱、公告板。
- 私聊/DM 协议骨架和 UI 初版。
- 头顶表情在主城、房屋、小游戏中统一表现。
- 聊天风控基础：节流、系统公告、举报入口预留。
- 主城移动同步做延迟、断线、重连测试。

### Phase 2：房屋与金币消耗（Week 3）

目标：建立归属感和金币消耗。

- 金币购买家具。
- 房屋布局保存、加载、编辑。
- 好友访问房屋只读模式。
- 房屋频道接入聊天。
- 家具配置与 Image 2 家具资产登记。

### Phase 3：钓鱼正式样板（Week 4）

目标：让第一个小游戏成为正式样板。

- 钓鱼体验打磨：动画、表情、奖励反馈、失败反馈。
- 服务端每日上限、结算账本、反作弊校验。
- 经济看板数据：金币产出、金币消耗、钓鱼结算、家具购买。
- 每日任务只围绕聊天、钓鱼、访问房屋、小小游戏完成。

### Phase 4：游戏大厅与创作者 Alpha（Week 5）

目标：内部可上传、可审、可白名单上线。

- 游戏大厅 v1：官方钓鱼入口 + 内部白名单小游戏列表。
- 创作者包上传 API。
- manifest 和资源扫描。
- GDScript 安全扫描规则。
- 异步审核状态：submitted、scanning、needs_review、approved、rejected、published。
- 管理员发布到小游戏大厅的内部入口。

### Phase 5：移动端内测准备（Week 6）

目标：TestFlight / Google Play 内测级别。

- Guest + Apple/Google 登录落地。
- 960x540 横屏移动体验打磨，375px 宽度压力测试。
- iOS/Android 构建流程和权限清单。
- Ubuntu 26.04 LTS 部署脚本、systemd、env、备份策略。
- QA 回归：聊天、移动、房屋、钓鱼、经济、审核、断线重连。

### Phase 6：小范围真实玩家测试（Week 8-12）

目标：验证留存、社交、经济和技术稳定性。

- 第二官方小游戏原型优先选挖矿 v0，塔防作为后续创作者示例方向。
- 好友列表、房屋访问、日常任务、更多家具和表情扩展。
- 小规模真实玩家测试：在线人数、聊天活跃、金币收支、断线重连、审核误杀率。
- 根据指标决定是否扩大内测或继续打磨主城循环。

## 关键指标

首批指标只服务产品判断，不追求复杂 BI：

- 每日会话时长：目标 20-60 分钟。
- 每局钓鱼完成率。
- 每日聊天消息数和表情使用次数。
- 房屋访问次数和家具购买次数。
- 金币每日净产出 / 净消耗，目标净产出不超过净消耗的 110%。
- 小游戏启动数、完成数、异常退出数。
- 创作者包校验通过率、拒绝原因分布。
- WebSocket 断线率、重连成功率、平均在线房间人数。

## 风险清单

### 最大产品风险

功能很多，但玩家没有日常理由回来。

缓解：

- 主城聊天、表情、房屋、钓鱼必须先组成 10 分钟可重复循环。
- 每个新系统必须回答“玩家今天为什么会用它”。

### 最大工程风险

UGC 动态加载和在线同步一起推进，容易产生不可控复杂度。

缓解：

- 创作者平台只做接口骨架和白名单 alpha。
- 官方钓鱼先当唯一完整样板。
- 后端所有结算和经济变更必须服务端权威。

### 最大美术风险

Image 2 生成资产风格漂移，导致主城、UI、角色、表情不像同一个产品。

缓解：

- 固定森林/RO-like 基底和 32 色以内调色约束。
- 每张 sheet 都先登记、切图、预览，再接入运行时。
- SVG 只保留临时配置合同，不进入正式主城/UI/房屋/小游戏。

### 最大运营风险

金币通胀和创作者收益规则过早复杂化。

缓解：

- 加速 MVP 期先只跟踪金币产出/消耗和家具 sink。
- 创作者收益先记录账本，不开放复杂提现或二级市场。

### 最大审核风险

玩家上传小游戏可能触发平台安全、内容和 App Store 审核风险。

缓解：

- MVP 不开放公开上传，只做内部白名单。
- 禁外链、禁广告、禁系统 API、禁访问主城节点。
- 审核异步，不阻塞上传流程。

## 近期执行队列

1. 主城 v1：用 Image 2 正式资产替换热点 blockout，保留交互逻辑。
2. 聊天 v1：补齐 DM 协议骨架、频道权限、节流和系统消息。
3. 房屋 v1：金币购买家具、布局保存、房屋频道接入。
4. 钓鱼 v1：结算反馈、每日上限、经济账本和 UI polish。
5. 创作者 alpha：manifest 校验与安全扫描对齐新版规范。
6. 后端运维：Ubuntu 26.04 LTS systemd/env/backup 文档和健康检查。
7. QA 体检：断线重连、重复结算、频道串线、房屋权限、资源缺失。

## 决策记录

- MVP 主城继续用当前森林/温暖小镇方向；港口城延后作为钓鱼区或 V1.1/V1.2 扩展。
- 不引入 C#、Java、Node.js 主服务。
- 不把完整 UGC 商店纳入首轮 12 周加速 MVP。
- 所有正式 UI/美术资产必须是 Image 2 生成后进入仓库的 PNG/WebP。
- 创作者小游戏必须支持单人模式，结算必须走 `on_end()`，发布必须经过服务端审核。
