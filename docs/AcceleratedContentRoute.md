# Accelerated Content Route

## 目标

这份路线用于替代“慢速 6 个月内容展开”的默认节奏。

核心判断：

- AI Studio 可以显著压缩文档、资产、代码、测试和配置迭代时间。
- 但线上安全、移动端适配、审核、经济稳定和真实玩家反馈不能无限压缩。
- 因此策略不是盲目堆功能，而是把内容扩展拆成可并行、可回滚、可测试的内容包。

## 加速原则

### 1. 先做内容管线，再堆内容

所有新内容必须走同一套入口：

- 配置：JSON / CSV / `.po`
- 资源：Image 2 PNG/WebP，登记到 `configs/ui_assets.json` 或 `configs/art_assets.json`
- 场景：Godot route / hotspot / catalog / minigame manifest
- 后端：REST / WebSocket contract
- 测试：content validation、Godot smoke、backend e2e、H5 viewport smoke

只有这样，后续新增 NPC、家具、表情、小游戏、活动才不会变成硬编码。

### 2. 内容扩展分五条并行线

1. 主城线：热点、NPC、建筑、场景氛围。
2. 社交线：聊天、表情、好友、房屋访问。
3. 生活线：钓鱼、采集、挖矿、日常任务。
4. 房屋线：家具、布局、风格、金币消耗。
5. 创作者线：`IMinigame`、manifest、安全扫描、白名单发布。

每条线可以并行推进，但每次合入必须保持主城可进入、聊天可用、钓鱼可玩、房屋可保存。

### 3. 每个内容包都要小而完整

一个内容包必须包含：

- 运行时入口。
- 三语 key。
- Image 2 正式资产或明确占位合同。
- 配置登记。
- 最少一个 smoke/e2e 覆盖点。
- 经济影响说明。

不允许只做“看起来有了”的半截内容。

## 4-6 周加速路线

### Week 0-1：主城核心锁定

目标：让当前森林主城成为真正的在线社交起点。

- 用 Image 2 正式建筑/道具替换主城 blockout 热点。
- 主城入口稳定：钓鱼、游戏大厅、房屋、商店。
- HUD 稳定：成员列表、聊天面板、小游戏入口、presence 心跳可视化。
- 表情统一：本地玩家、远端玩家、小游戏都走头顶气泡。
- H5 landscape 可测，portrait 继续显示横屏提示。

验收：

- 两个玩家进入同一房间，可以看到 presence、聊天、表情。
- 主城四个热点都能触发正确入口或系统提示。
- Web/H5 smoke 通过。

### Week 2：内容包 1 - 热闹小镇

目标：让主城从“功能图”变成“有人味的地方”。

- NPC 第一批：渔夫、商人、邮箱员、小游戏主持、房屋管理员、活动公告员。
- 主城氛围：灯、花、路牌、摊位、邮箱、公告板。
- 聊天增强：频道历史、系统公告样式、基础节流。
- 表情快捷：保留 Alt 快捷键，并准备移动端表情轮盘。
- 日常任务 v0：聊天一次、钓鱼一次、访问房屋一次。

验收：

- 主城首屏能明确看到“社交、钓鱼、房屋、小游戏”的视觉入口。
- 所有新增可见文字走 localization key。
- 新增资产全部登记。

### Week 3：内容包 2 - 房屋与金币消耗

目标：让玩家有第一个长期投入点。

- 家具商店 v1：用金币购买家具。
- 家具包 v1：床、桌、椅、植物、地毯、灯、收纳、小游戏机。
- 房屋访问 v0：好友/指定玩家只读访问。
- 房屋聊天频道接入。
- 金币消耗记录：家具购买、摆放、墙地样式切换。

验收：

- 玩家可以从主城进入家，购买家具，摆放后保存。
- 后端拒绝越权修改他人房屋。
- 经济账本能区分钓鱼收入和房屋消耗。

### Week 4：内容包 3 - 钓鱼正式样板

目标：钓鱼成为第一个能展示给创作者参考的完整官方小游戏。

- 钓鱼结果 UI polish：鱼名、重量、金币、稀有度、今日上限。
- 服务端可信结算：每日/每 session cap、幂等、防重复领取。
- 钓鱼素材包整理：湖边、鱼、浮漂、鱼竿、结果面板。
- 创作者示例文档：钓鱼如何继承 `IMinigame`，如何结算，如何请求表情。
- 经济看板 v0：收入、支出、净产出、上限触发次数。

验收：

- 钓鱼可以作为 creator reference package 交给 AI 读取。
- 重复提交奖励不会重复发金币。
- 今日上限触发后客户端有清晰反馈。

### Week 5：内容包 4 - 游戏大厅与创作者 Alpha

目标：小游戏平台从概念变成内部可验证流程。

- 游戏大厅 v1：官方钓鱼入口 + 内部白名单小游戏列表。
- 创作者包上传 API。
- manifest 校验：三语名称、版本、玩家数、资源大小、入口场景。
- GDScript 安全扫描：禁外链、禁系统 API、禁访问主城节点。
- 审核状态机：submitted、scanning、needs_review、approved、rejected、published。

验收：

- 一个内部 creator package 可以提交、扫描、审核、发布到游戏大厅。
- 审核失败能返回可读原因。
- 未审核小游戏不能被普通玩家启动。

### Week 6：移动内测准备

目标：从 H5 可测推进到移动内测候选。

- Guest + Apple/Google 登录路线对齐。
- 960x540 横屏移动 UI polish。
- 375px 宽度压力测试继续作为底线。
- iOS/Android 构建检查表。
- Ubuntu 26.04 LTS 部署检查：systemd、env、PostgreSQL、Redis、备份。
- QA 总回归：登录、聊天、移动、表情、房屋、钓鱼、小游戏审核、断线重连。

验收：

- H5、Godot headless、Go test/e2e 全绿。
- 移动端横屏主流程可录屏。
- 部署文档可按步骤复现。

## 内容扩展优先级

### P0：必须尽快扩

- 主城正式建筑和功能入口。
- NPC 第一批。
- 家具第一批。
- 钓鱼正式结果 UI。
- 表情移动端入口。
- 聊天历史和频道稳定。
- 经济账本和每日上限。

### P1：加速进入

- 好友列表 v0。
- 房屋访问 v0。
- 游戏大厅 v1。
- 创作者白名单 alpha。
- 第二个官方小游戏原型：优先挖矿或轻塔防。
- 每日任务 v0。

### P2：不要提前抢主线

- 多城市大地图。
- 宠物深养成。
- 公会。
- 玩家交易市场。
- 创作者收益提现。
- 大型 PVP 或复杂战斗职业。

## 第二官方小游戏建议

为了内容扩展速度，第二个官方小游戏建议二选一：

### 方案 A：挖矿

优点：

- 与森林/生活技能契合。
- 美术需求少，可复用洞穴、矿石、镐子、结果面板。
- 经济上适合作为金币和家具材料来源。

风险：

- 容易变成无聊点击，需要手感和随机事件。

### 方案 B：轻塔防

优点：

- 更像“小游戏大厅”的展示内容。
- 容易做排行榜和多人观战。

风险：

- 平衡、路径、怪物波次、UI 都更重。

当前建议：

> 先做挖矿 v0，塔防保留为创作者示例方向。

## 加速后的阶段目标

- 2 周：主城在线社交核心可持续测试。
- 4 周：房屋、钓鱼、金币闭环进入 v1。
- 6 周：游戏大厅 + 创作者 alpha + 移动内测准备。
- 8-10 周：第二官方小游戏、好友访问、活动日常、更多家具和表情。
- 12 周：小范围真实玩家测试。

## 不被加速压垮的底线

- 任何可见文字都必须有 localization key。
- 任何正式 UI/美术都必须来自 Image 2 PNG/WebP。
- 任何金币变动都必须服务端权威或可被服务端对账。
- 任何 creator minigame 都必须通过 manifest 和安全扫描。
- 任何新内容都不能破坏主城进入、聊天、房屋保存、钓鱼结算。
